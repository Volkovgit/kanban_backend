# Kanban Backend API - Документация для Frontend-разработчика

В данном документе описано взаимодействие клиентской части с Kanban Backend API. Внимательно ознакомьтесь с правилами авторизации, стандартным форматом ответов, обработкой ошибок и списком контрактов.

---

## 📌 Базовый URL
Все запросы к API должны выполняться по префиксу:
```
/api/v1
```

---

## 🛡️ Аутентификация и Токены (JWT)

API использует JWT (JSON Web Tokens) для защиты роутов. 
1. После успешного входа (`/auth/login`) или регистрации (`/auth/register`) вы получите пару токенов: **`accessToken`** и **`refreshToken`**.
2. **`accessToken`** (время жизни: 1 час) необходимо передавать в заголовке `Authorization` при каждом запросе к защищенным эндпоинтам:
   ```http
   Authorization: Bearer <твой_access_token>
   ```
3. Когда `accessToken` истекает, сервер вернет **401 Unauthorized**. Вам нужно отправить запрос на `/auth/refresh` с текущим **`refreshToken`** в теле запроса, чтобы получить новую пару токенов (ротация токенов).
4. **`refreshToken`** (время жизни: 30 дней) используется *только* для получения новых пар токенов и для выхода (`/auth/logout`). 

---

## 📦 Общий формат ответов (Contracts)

### 🟢 Успешный ответ (200 OK, 201 Created)
Успешные запросы всегда возвращают объект со свойством `success: true` и полезной нагрузкой в `data`. Дополнительно может приходить поле `message`.

```json
{
  "success": true,
  "data": { ... },
  "message": "Опциональное сообщение (например, 'Успешный вход')"
}
```

### 🔴 Ответ с ошибкой (4xx, 5xx)
Ошибки всегда возвращают стабильную структуру:

```json
{
  "success": false,
  "error": {
    "code": "BAD_REQUEST",    // Строковый код ошибки (VALIDATION_ERROR, UNAUTHORIZED, NOT_FOUND и т.д.)
    "message": "Текст ошибки", // Описание человеческим языком
    "errors": [               // Присутствует только при ошибках валидации
      {
        "field": "title",
        "constraints": ["title must be a string", "title cannot be empty"]
      }
    ]
  },
  "timestamp": "2024-02-24T12:00:00.000Z",
  "path": "/api/v1/auth/login",
  "correlationId": "опционально-uuid-для-трейсинга"
}
```

---

## 🚦 Ограничение запросов (Rate Limits)
- **Auth endpoints (`/auth/*`)**: Строгий лимит — 10 запросов в минуту с одного IP-адреса для защиты от брутфорса. При превышении возвращается код `429 Too Many Requests`.
- **Остальные API endpoints**: 100 запросов в минуту для авторизованного пользователя.

Также присутствует механизм блокировки аккаунта: **5 неудачных попыток входа подряд** временно блокируют аккаунт на 15 минут.

## 📚 Справочник Эндпоинтов

### 1. 🔑 Авторизация (Auth)

#### `POST /auth/register`
Регистрация нового пользователя.
- **Тело запроса:**
  ```json
  {
    "login": "user@example.com", // Строка, уникальный логин (до 255 символов)
    "password": "strongPassword123" // Строка, минимум 6 символов, максимум 100
  }
  ```
- **Ответ (201):** `data: { "id": "uuid", "login": "user@example.com" }`

#### `POST /auth/login`
Вход в систему.
- **Тело запроса:**
  ```json
  {
    "login": "user@example.com",
    "password": "strongPassword123"
  }
  ```
- **Ответ (200):** `data: { "accessToken": "jwt...", "refreshToken": "jwt..." }`

#### `POST /auth/refresh`
Обновление токенов.
- **Тело запроса:**
  ```json
  {
    "refreshToken": "твой_текущий_refresh_token"
  }
  ```
- **Ответ (200):** `data: { "accessToken": "новый_jwt...", "refreshToken": "новый_jwt..." }`

#### `POST /auth/logout`
Выход из системы.
- **Заголовки:** `Authorization: Bearer <access_token>`
- **Тело запроса:**
  ```json
  {
    "refreshToken": "твой_текущий_refresh_token"
  }
  ```
- **Ответ (200):** `success: true`

### 2. 🗂️ Доски (Boards)
*Все эти роуты требуют заголовок `Authorization: Bearer <accessToken>`.*

Данные досок изолированы: вы получаете только те доски, которые создали сами.

#### `GET /boards`
Получение списка ваших досок с пагинацией.
- **Query параметры (опционально):** `?page=1&pageSize=10`
- **Ответ (200):**
  ```json
  {
    "success": true,
    "data": [
      {
        "id": "uuid",
        "title": "Рабочий проект",
        "description": "Описание доски",
        "ownerId": "uuid",
        "createdAt": "2024...",
        "updatedAt": "2024..."
      }
    ],
    "pagination": { "page": 1, "pageSize": 10, "totalCount": 1, "totalPages": 1 }
  }
  ```

#### `GET /boards/:id`
Получение одной доски по ID (ID должен быть формата UUID).
- **Ответ (200):** Объект профиля доски в `data`. Возвращает `404` если не найдено.

#### `POST /boards`
Создание новой доски.
- **Тело запроса:**
  ```json
  {
    "title": "Моя новая доска", // Строка, обязательно, макс. 100 символов
    "description": "Опционально" // Строка, опционально, макс. 500 символов
  }
  ```
- **Ответ (201):** Возвращает созданный объект доски.

#### `PUT /boards/:id`
Обновление доски.
- **Тело запроса (все поля опциональны):**
  ```json
  {
    "title": "Новое название",
    "description": "Новое описание"
  }
  ```
- **Ответ (200):** Обновленный объект.

#### `DELETE /boards/:id`
Удаление доски (вместе со всеми её задачами).
- **Ответ (204 No Content):** Успех. Нет тела ответа.

### 3. ✅ Задачи (Tasks)
*Все эти роуты требуют заголовок `Authorization: Bearer <accessToken>`.*

Задачи всегда привязаны к конкретной доске.

#### `GET /boards/:boardId/tasks`
Получение задач для доски (с фильтрацией и пагинацией).
- **Параметр пути:** `boardId` (Обязательно, UUID).
- **Query параметры (все опциональны):** 
  - `page` (по умолчанию 1)
  - `pageSize` (по умолчанию 10, макс 100)
  - `status` (поиск по статусу: `todo` | `in_progress` | `done`)
  - `priority` (поиск по приоритету: `low` | `medium` | `high`)
- **Ответ (200):** Возвращаются задачи внутри ключа `data` и информация в `pagination`.

#### `POST /boards/:boardId/tasks`
Создание задачи в доске.
- **Тело запроса:**
  ```json
  {
    "title": "Купить молоко", // Обязательно, до 200 симв.
    "description": "Описание задачи", // Опционально, до 2000 симв.
    "status": "todo", // Опционально. Значения: 'todo', 'in_progress', 'done'. По умолчанию: 'todo'
    "priority": "medium" // Опционально. Значения: 'low', 'medium', 'high'. По умолчанию: 'medium'
  }
  ```
- **Ответ (201):** Созданная задача.

#### `PUT /boards/:boardId/tasks/:taskId`
Обновление задачи (перемещение по статусам, смена приоритета, текста).
- **Параметры пути:** `boardId` и `taskId` (строгие UUID).
- **Тело запроса (разрешены частичные обновления, передавайте только то, что изменилось):**
  ```json
  {
    "title": "Новое имя",
    "description": "Новый текст",
    "status": "in_progress", // Значения: 'todo', 'in_progress', 'done'
    "priority": "high" // Значения: 'low', 'medium', 'high'
  }
  ```
- **Ответ (200):** Обновленная задача. 
- *Примечание: Задачи можно перемещать из любого `status` в любой другой.*

#### `DELETE /boards/:boardId/tasks/:taskId`
Удаление задачи.
- **Ответ (204 No Content):** Успех. Нет тела ответа.

### 4. 🩺 Системные (Health)

#### `GET /health`
Публичный эндпоинт для проверки статуса бекенда (JWT не требуется).
- **Ответ (200):**
  ```json
  {
    "success": true,
    "data": {
      "status": "ok",
      "timestamp": "2024-02-24T12:00:00.000Z",
      "uptime": 120.45,
      "database": "connected",
      "version": "1.0.0"
    }
  }
  ```

## 🛠 Краткая справка по типам Enums (TypeScript)

Если вы используете TypeScript на клиенте, вам пригодятся эти энумы:

```ts
export enum TaskStatus {
  TODO = 'todo',
  IN_PROGRESS = 'in_progress',
  DONE = 'done',
}

export enum TaskPriority {
  LOW = 'low',
  MEDIUM = 'medium',
  HIGH = 'high',
}
```

Все ID (`userId`, `boardId`, `taskId`) генерируются на сервере как `UUID v4`. Обязательно сохраняйте и передавайте их в строковом формате.
